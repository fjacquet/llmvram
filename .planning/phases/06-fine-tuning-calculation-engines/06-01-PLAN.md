3. `TrainingPrecision` union type: `'fp32' | 'fp16' | 'bf16'`

4. `TrainingVRAMBreakdown` interface with Decimal fields:
   - `modelWeights` — Model weight storage at training precision (or quantized for LoRA/QLoRA frozen base)
   - `masterWeights` — FP32 master copy for mixed precision (zero if fp32 training)
   - `gradients` — Gradient storage (same precision as training weights, only for trainable params)
   - `optimizerStates` — Optimizer state storage (ALWAYS FP32, see PITFALLS.md #2)
   - `activations` — Training activation memory (batch-dependent)
   - `frameworkOverhead` — Framework overhead (Decimal)
   - `total` — Sum of all components (Decimal)
   - `trainableParameters` — Count of trainable parameters in billions (Decimal)
   - `totalParameters` — Count of total parameters in billions (Decimal)
   - `method` — FineTuningMethod used

5. `LoRAVRAMBreakdown` interface extending TrainingVRAMBreakdown with:
   - `baseWeights` — Frozen base model weights (Decimal)
   - `adapterWeights` — LoRA adapter weights in FP16 (Decimal)
   - `adapterParameters` — Adapter parameter count (Decimal)

All interfaces must use `import type Decimal from 'decimal.js'` (already imported in file).
Add JSDoc comments referencing research (06-RESEARCH.md) and PITFALLS.md for critical fields like optimizerStates.

**In `src/engines/constants.ts`, add the following constants AFTER existing constants:**

1. `OPTIMIZER_STATE_BYTES` — `Record<OptimizerType, Decimal>` mapping:
   - `'adamw'`: `new Decimal(8)` — 2 FP32 states (momentum + variance) = 8 bytes/param
   - `'sgd-momentum'`: `new Decimal(4)` — 1 FP32 state (momentum) = 4 bytes/param
   - `'adamw-8bit'`: `new Decimal(2)` — Quantized 8-bit states = 2 bytes/param
   - `'adafactor'`: `new Decimal(4)` — Factored approximation states = 4 bytes/param

2. `TRAINING_FRAMEWORK_OVERHEAD_GB` — `new Decimal(1.5)` — Training uses more overhead than inference (1.5GB vs 1.0GB) due to autograd graph, gradient buffers, and optimizer state management.

3. `TOTAL_TARGETABLE_MODULES_PER_LAYER` — `7` as a plain number constant — Standard transformer layer has 7 targetable linear modules: q_proj, k_proj, v_proj, o_proj (attention) + gate_proj, up_proj, down_proj (MLP).

4. `GRADIENT_BYTES` — `Record<TrainingPrecision, Decimal>` mapping:
   - `'fp32'`: `new Decimal(4)`
   - `'fp16'`: `new Decimal(2)`
   - `'bf16'`: `new Decimal(2)`

5. `WEIGHT_BYTES` — `Record<TrainingPrecision, Decimal>` mapping (same values as GRADIENT_BYTES, but semantically distinct for clarity):
   - `'fp32'`: `new Decimal(4)`
   - `'fp16'`: `new Decimal(2)`
   - `'bf16'`: `new Decimal(2)`

6. `MASTER_WEIGHT_BYTES` — `new Decimal(4)` — FP32 master weights for mixed precision training.

Import `OptimizerType` and `TrainingPrecision` from `'./types'` in constants.ts. Follow existing JSDoc patterns with research references.

**Add a `TrainingInputSchema` Zod schema AFTER the existing `validateModels` function in `src/utils/schemas.ts`:**

```typescript
// Training configuration schema for fine-tuning VRAM calculation
export const TrainingInputSchema = z.object({
  /** Fine-tuning method */
  method: z.enum(['full', 'lora', 'qlora']),

  /** Optimizer type — affects optimizer state memory */
  optimizer: z.enum(['adamw', 'sgd-momentum', 'adamw-8bit', 'adafactor']),

  /** Training precision — affects weight and gradient memory */
  trainingPrecision: z.enum(['fp32', 'fp16', 'bf16']),

  /** Training batch size (micro-batch per GPU) */
  batchSize: z.number().int().min(1).max(128),

  /** Sequence length for training */
  sequenceLength: z.number().int().min(512).max(131072),

  /** LoRA rank — controls adapter capacity (only used for lora/qlora methods) */
  loraRank: z.number().int().min(4).max(256).default(16),

  /** LoRA alpha — scaling factor (alpha/rank), does NOT affect VRAM */
  loraAlpha: z.number().int().min(1).max(512).default(32),

  /** Percentage of linear modules per layer that get LoRA adapters (10-100%) */
  targetModulesPercent: z.number().int().min(10).max(100).default(30),
})
```

Also add:

1. Inferred type: `export type TrainingInput = z.infer<typeof TrainingInputSchema>`

2. Validation helper: `export function validateTrainingInput(data: unknown): TrainingInput { return TrainingInputSchema.parse(data) }`

Follow the existing pattern in schemas.ts (GPUSchema/ModelSchema have matching validate functions and type exports).
