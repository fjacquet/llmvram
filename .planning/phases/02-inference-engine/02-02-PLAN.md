<objective>
Build the core VRAM calculation engines: KV cache with GQA/MQA support and the main inference VRAM calculator that combines weights, KV cache, activations, and framework overhead.

Purpose: This is the heart of the application — the calculation that answers "how much VRAM does this model need on this hardware?" Getting the KV cache GQA ratio and MoE parameter handling right is critical for accuracy. A wrong KV cache estimate (ignoring GQA) overestimates by 2-8x. A wrong MoE estimate (using active instead of total params) underestimates by 3-4x.

Output: Two fully tested engines (kv-cache.ts and inference.ts) that produce accurate VRAM breakdowns for any model/quantization/sequence-length combination.
</objective>

Create `src/engines/kv-cache.ts` with one pure function:

**calculateKVCacheVRAM(params: { model: Model, sequenceLength: number, batchSize: number, kvPrecision: KVCachePrecision }): Decimal**

Formula (from NVIDIA documentation):
```
KV_bytes = 2 * num_hidden_layers * hidden_size * sequenceLength * batchSize * precisionBytes * gqaRatio
```

Where:
- `2` = Key + Value tensors
- `gqaRatio = (model.num_kv_heads ?? model.num_attention_heads) / model.num_attention_heads`
  - Standard MHA: ratio = 1.0 (num_kv_heads equals num_attention_heads)
  - GQA (Llama 3 70B): ratio = 8/64 = 0.125
  - MQA: ratio = 1/num_attention_heads (single KV head)
- `precisionBytes` = lookup from KV_PRECISION_BYTES constant

Return `KV_bytes / BYTES_PER_GB` as Decimal (in GB).

Use Decimal.js for ALL arithmetic. Import Model type from `@utils/schemas`. Import KVCachePrecision from `@engines/types`. Import KV_PRECISION_BYTES and BYTES_PER_GB from `@engines/constants`.

Create `src/engines/kv-cache.test.ts`:

Then create same model but with num_kv_heads=64 (MHA equivalent). The MHA result divided by GQA result should equal 8.0 (8x reduction).

   Expected = 2 * 80 * 8192 * 4096 * 1 * 2 * (8/64) / 1024^3 = ~0.9375 GB

  <action>
Create `src/engines/inference.ts` with these pure functions:

1. **calculateActivationMemory(model: Model, sequenceLength: number, batchSize: number): Decimal**
   - For dense models: `batchSize * sequenceLength * intermediateSize * 4 / BYTES_PER_GB`
   - The `* 4` is for FP32 activation storage
   - For MoE models: use active parameters to scale. Call calculateMoEActiveParams() to get the effective intermediate_size contribution.
   - This is a simplified formula acceptable for Phase 2 (per research). Phase 3 can refine.

2. **calculateMoEActiveParams(model: Model): number**
   - If model is not MoE (no num_experts or num_experts_per_token), return model.num_parameters_billion
   - For MoE: `sharedParams + (expertParams * activeRatio)` where:
     - `activeRatio = num_experts_per_token / num_experts`
     - `expertParams = num_parameters_billion * 0.8` (approx 80% in expert layers)
     - `sharedParams = num_parameters_billion * 0.2` (approx 20% shared)
   - Returns number (billions), used only for activation sizing, NOT for weight VRAM

3. **calculateInferenceVRAM(params: { model: Model, quantization: QuantizationFormat, sequenceLength: number, batchSize: number, kvQuantization?: KVCachePrecision }): InferenceVRAMBreakdown**
   - `modelWeights` = calculateModelWeightVRAM(model.num_parameters_billion, quantization)
     - CRITICAL: Uses model.num_parameters_billion directly. For MoE this is TOTAL params (46.7B for Mixtral). The model database already stores total params per Phase 1 decision.
   - `kvCache` = calculateKVCacheVRAM({ model, sequenceLength, batchSize, kvPrecision: kvQuantization ?? 'fp16' })
   - `activations` = calculateActivationMemory(model, sequenceLength, batchSize)
   - `frameworkOverhead` = FRAMEWORK_OVERHEAD_GB (1.0 GB constant)
   - `total` = modelWeights + kvCache + activations + frameworkOverhead
   - Return InferenceVRAMBreakdown with all fields as Decimal

Import calculateModelWeightVRAM from quantization.ts. Import calculateKVCacheVRAM from kv-cache.ts. Import types from types.ts. Import constants from constants.ts.
